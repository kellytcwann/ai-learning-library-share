# Insurance POS Basics

這頁解釋保險 POS / sales platform 的基本概念，適合初次了解 POS 時先建立共同語言。

## 概念

保險 POS（Point of Sale）通常指銷售前台平台：幫 agent / adviser 把客戶對話變成 needs analysis、quote / illustration、application、文件提交和 case tracking。

在 target insurance client 脈絡下，已知他們使用「current POS / sales platform」，但 current POS / sales platform 的實際範圍仍要向 team / client 確認。

## 為什麼對保險轉型重要

POS 是保險銷售旅程的入口。如果 POS 設計不好，會造成重複輸入、quote 解釋困難、文件錯漏、NIGO、manual referral、產品上架慢，最後影響銷售體驗和營運效率。

## 典型 Journey

| 步驟 | 例子 | 讀者要觀察 |
|---|---|---|
| Needs analysis | agent 收集保障、儲蓄、健康需求 | 是否有標準流程和 audit trail？ |
| Product selection | 選 base plan 和 riders | 規則是系統判斷還是人工理解？ |
| Illustration | 產生 premium / benefits projection | 客戶是否容易理解？ |
| Application | 填 proposal / e-application | 資料是否重複輸入？ |
| Document upload | 上載 KYC、醫療、財務文件 | 是否提交前已檢查完整性？ |
| Submission | 送 underwriting / operations | NIGO 和 referral 的原因是什麼？ |

## Key Terms / 關鍵詞

| 專業詞 | 中文 / 說明 |
|---|---|
| FNA / needs analysis | 客戶需求分析，支持產品建議和 suitability。 |
| Illustration | 保費、保障、現金價值或利益示例。 |
| Rider | 附加保障。 |
| Suitability | 產品是否適合客戶需要和情況。 |
| NIGO | Not in good order，申請資料不完整或有錯。 |
| STP | Straight-through processing，無需人工返工即可處理。 |
| Referral | 因規則、風險或例外而轉人工審批。 |

## BA / PM Relevance

| Lens | 讀者要做什麼 |
|---|---|
| BA | 拆 journey、記錄欄位、規則、痛點、控制點。 |
| PM | 確認 scope、stakeholder、timeline、交付物和依賴。 |
| FDE | 看資料在哪裡產生、如何流向 downstream systems、哪些步驟可自動化。 |

## 讀者可以問

- Which steps are actually performed inside current POS / sales platform?
- Which steps happen outside current POS / sales platform, such as email, Excel, CRM, or manual review?
- What are the top reasons for NIGO or referral?
- Which product rules are validated before submission?
- What evidence do we have: screenshots, process map, case logs, user feedback?

## Meeting-Ready Wording

> 「我想 confirm 一下，今次 current POS / sales platform scope 係咪 cover quote、illustration、application capture、document upload 同 submission，定只係其中一部分？」
